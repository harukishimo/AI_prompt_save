# 完了定義: 駄菓子事業向け営業支援ツール開発

## 目的

駄菓子事業の事業計画を調査・整理し、その営業活動に必要な知識を元に、営業効率と対応スピードを上げる営業支援ツールを開発する。ツールは現在のRAGアプリケーションとdesignを土台にして、営業現場で実際に使える状態まで仕上げる。

## 完了状態

この取り組みは、以下の成果物がそろい、受け入れ基準を満たした時点で完了とする。

1. 駄菓子事業の事業計画調査が完了している
2. 営業活動に必要な知識が整理され、RAGで参照できる状態になっている
3. 営業支援ツールの主要機能が実装されている
4. 現在のRAGアプリケーションとdesignが反映されている
5. 営業担当者が実務で使える品質まで検証されている

## 1. 事業計画調査の完了定義

駄菓子事業について、以下が文書化されていること。

- 事業概要: 誰に、何を、どの販売チャネルで提供する事業か
- 顧客セグメント: 小売店、イベント事業者、学校・地域団体、企業ノベルティ、EC購入者などの分類
- 提供価値: 価格、懐かしさ、詰め合わせ、イベント性、地域性、ギフト性など
- 競合調査: 直接競合、間接競合、代替手段、価格帯、差別化ポイント
- 販売チャネル: 店舗営業、卸、EC、イベント、法人営業、SNS経由など
- 収益構造: 仕入れ、粗利、送料、在庫、廃棄、販促費、人件費の考え方
- 営業戦略: 初回接点、提案内容、商談フロー、リピート獲得方法
- リスク: 食品表示、賞味期限、保管、在庫過多、季節変動、取引条件
- KPI: 商談数、成約率、平均受注額、リピート率、粗利率、在庫回転率
- 情報源一覧: 参照した資料、Webページ、社内資料、ヒアリング記録

受け入れ基準:

- 調査内容が営業活動に使える粒度で整理されている
- 重要な主張には根拠または参照元がある
- 不明点、仮説、追加調査が必要な項目が分離されている
- 事業計画の内容から営業トーク、提案資料、FAQに展開できる

## 2. 営業知識整理の完了定義

営業支援ツールに投入する知識として、以下が整理されていること。

- 商品知識: 商品カテゴリ、価格帯、原価・粗利の考え方、賞味期限、保管条件
- 顧客別提案パターン: 小売、イベント、法人、地域団体、EC向けの提案例
- 営業トーク: 初回連絡、課題ヒアリング、提案、クロージング、フォロー
- 反論対応: 価格、在庫、納期、賞味期限、競合比較、取引条件への回答
- FAQ: 顧客から聞かれる質問と回答
- 提案テンプレート: メール文面、商談メモ、見積もり前提、提案要約
- 営業フロー: リード獲得から受注・再提案までのステータス
- 禁止・注意表現: 誇大表現、根拠のない効能、食品関連で避けるべき表現

受け入れ基準:

- RAGで検索・回答に使いやすい単位に分割されている
- 各ナレッジにタイトル、カテゴリ、対象顧客、更新日、出典がある
- 回答生成時に参照元を表示できる
- 営業担当者がそのまま使える文面と、社内向けの補足が分かれている

## 3. 営業支援ツールの機能完了定義

最低限、以下の機能が動作すること。

- ナレッジ検索: 駄菓子事業、商品、顧客、営業トーク、FAQを横断検索できる
- RAG回答: 営業質問に対して、根拠付きで回答を生成できる
- 顧客別提案生成: 顧客タイプや課題に応じた提案文を作成できる
- メール・DM文面生成: 初回営業、フォロー、再提案の文面を作成できる
- 反論対応支援: 顧客の懸念に対する回答候補を提示できる
- 商談メモ作成: ヒアリング内容から要点、次回アクション、提案方針を整理できる
- 営業フロー管理: リード、接触済み、商談中、見積もり、受注、失注などの状態を扱える
- 履歴参照: 過去の質問、生成文面、商談メモを確認できる

優先度の高い拡張機能:

- 顧客情報の登録・編集
- 提案テンプレートの保存
- ナレッジの追加・更新
- 回答品質へのフィードバック
- CSVインポート・エクスポート
- CRM連携を想定したデータ構造

受け入れ基準:

- 営業担当者が、調査なしで初回営業文面を作れる
- 顧客課題を入力すると、提案の方向性と根拠が出る
- 回答には参照元が表示され、根拠不明な断定を避ける
- 生成結果は編集可能で、そのまま送信前の下書きとして使える
- 入力不足の場合は、追加で聞くべき質問を提示する

## 4. 現在のRAGアプリケーション反映の完了定義

既存RAGアプリケーションを土台にする場合、以下が満たされていること。

- 既存の検索・回答生成フローを確認している
- 既存のデータ投入方法、チャンク分割、埋め込み、検索、プロンプトを把握している
- 駄菓子事業向けナレッジを投入できる
- 営業用途に合わせたシステムプロンプトまたは回答ルールが設定されている
- 参照元表示、回答不能時の挙動、曖昧な質問への確認質問が実装されている
- 既存機能を壊さず、営業支援用途の画面またはモードを追加している

受け入れ基準:

- 既存RAGアプリケーションの主要機能が回帰テストで動く
- 営業支援用の質問で、関連ナレッジが検索結果に出る
- 回答がナレッジに基づいており、根拠のない補完が抑制されている
- ナレッジ更新後に検索・回答へ反映できる

## 5. design反映の完了定義

現在のdesignを元に、以下が実装されていること。

- 既存designのレイアウト、色、余白、文字サイズ、コンポーネント規則を確認している
- 営業支援ツールの主要画面がdesignに沿っている
- ナレッジ検索、回答、提案生成、履歴、顧客情報の画面導線が自然である
- PCでの営業作業を優先しつつ、必要に応じてモバイルでも破綻しない
- ローディング、エラー、空状態、入力不足、保存完了などの状態が用意されている

受け入れ基準:

- 既存designと見た目・操作感が大きくずれていない
- 主要操作が3クリック以内、または1画面内で完了できる
- 長い回答、長い顧客名、長い商品名でもレイアウトが崩れない
- 営業担当者が、検索、生成、編集、保存、履歴確認を迷わず行える

## 6. 品質・検証の完了定義

以下の確認が完了していること。

- 代表的な営業シナリオで動作確認している
- RAG回答の根拠表示を確認している
- 存在しない情報を聞いた場合に、推測で断定しないことを確認している
- 入力不足時に確認質問または不足項目が出る
- 生成文面が営業で使える自然な日本語になっている
- 食品・価格・納期・条件に関する断定が、根拠なしに出ない
- 主要画面の表示崩れがない
- エラー時にユーザーが次の行動を取れる

テスト観点:

- ナレッジ検索精度
- 回答生成品質
- 参照元表示
- 顧客別提案生成
- 反論対応
- 商談メモ生成
- 履歴保存
- ナレッジ更新
- 画面表示
- 権限・データ保護

## 7. 最終受け入れシナリオ

以下のシナリオが通れば、実務利用可能な最小完成版とする。

1. 「地域イベント向けに駄菓子詰め合わせを提案したい」と入力する
2. ツールが提案方針、想定商品、訴求点、確認すべき条件を出す
3. 回答に参照元が表示される
4. 初回営業メールの下書きを生成できる
5. 顧客の「価格が高い」という反論に対する回答候補を出せる
6. 商談メモとして要点と次回アクションを保存できる
7. 保存した履歴を後から確認できる
8. 既存designと整合した画面で操作できる

## 8. 未完了と判断する条件

以下のいずれかが残っている場合は未完了とする。

- 事業計画調査が営業活動に使える形で整理されていない
- ナレッジの出典や根拠が不明
- RAG回答が参照元なしで断定する
- 営業文面が一般論に偏り、駄菓子事業向けになっていない
- 既存RAGアプリケーションとの接続ができていない
- designとの整合が確認できていない
- 主要シナリオで操作が最後まで完了しない
- 保存、履歴、エラー処理など実務利用に必要な基本動作がない

## 9. 次に決めること

実装前に、以下を確定する。

- 既存RAGアプリケーションの技術構成
- 既存designの参照先
- 最初に対象とする顧客セグメント
- ナレッジ投入元の資料
- 営業支援ツールの最小機能範囲
- 保存する顧客情報と営業履歴の項目
- 利用者、権限、データ管理方針
