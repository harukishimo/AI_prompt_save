# 駄菓子事業計画エージェント運用 TODO

## 目的

ユーザーとCodexで壁打ちしながら、駄菓子事業の事業計画書をAIエージェント群で作成・検証・更新できる状態にする。

このTODOは、各エージェントのMarkdown定義、GitHub Issue運用、Looperによる自動処理、PRレビュー、ユーザー最終確認までの仕組みを整えるための作業一覧である。

## 全体方針

- Looperは処理フローの司令塔として扱う。
- 事業計画ディレクターは事業計画の方針・採用判断・統合判断を担う。
- 専門エージェント同士は相互レビューを行い、別観点から矛盾・補強・代替案を出す。
- 最終的な事業判断は事業計画ディレクターに集約する。
- 事業計画ライターは、ディレクターが確定した執筆指示書をもとに文章化する。
- 事業計画書の章構成は最初から固定せず、Issueを作成するための仮構成として扱う。
- 仮章構成をもとにGitHub Issueを切り、壁打ち・調査・設計・レビューを進めながら章構成を更新する。
- 最終的な事業計画書レイアウトは、ユーザー、事業計画ディレクター、事業計画ライターのコミュニケーションで確定する。
- 課金、購入、有料登録、契約、外部申請はユーザー確認なしに行わない。
- 法律、税務、食品衛生、許認可は断定せず、要確認事項として扱う。

### Issueの基本分類

- [ ] 章別Issue: 事業概要、想定顧客、収支計画など、事業計画書の各章を作成・検証するIssue。
- [ ] 横断Issue: 章同士の整合性、根拠管理、未決事項、レイアウト方針など、複数章にまたがるIssue。
- [ ] 壁打ちIssue: ユーザーとの対話から出た未決事項、仮説、追加質問を整理するIssue。
- [ ] レビューIssue: PRレビューや専門エージェントの指摘を再処理するIssue。
